# Phase 12c — Production Deployment Runbook (Simplified)

**Project:** XCloud
**Type:** No code changes — operational checklist
**Dependencies:** Phases 1–11, plus Phases 12a (Setup Wizard) and 12b (DB Backup)

> ✅ **What changed vs original Phase 12:** All phpMyAdmin steps are gone. SMTP setup, testing, smoke tests, and registration enable are all done in the Setup Wizard. Backups are one click in admin panel.

## What stays manual (cannot be automated with cPanel)

1. **DNS records** — done once, at your DNS provider
2. **File upload** — drag/drop in cPanel File Manager
3. **First-time admin email setup** — needed once if admin user doesn't have an email yet

Everything else is now **one-click in browser**.

---

## One-time pre-flight (do once, ever)

### A. DNS records on `xcloud.retrivo.ir`

```
SPF:    v=spf1 a mx ip4:<server-ip> -all
DKIM:   <generated by mail server, public key in DNS>
DMARC:  v=DMARC1; p=quarantine; rua=mailto:postmaster@retrivo.ir
PTR:    set on the server's IP via hosting provider
```

Verify after publish:
```bash
dig TXT xcloud.retrivo.ir +short
dig TXT _dmarc.xcloud.retrivo.ir +short
dig TXT default._domainkey.xcloud.retrivo.ir +short
```

### B. Admin email (one-time, only if admin user has no email)

If the admin user doesn't have an email set yet, log in as admin and use the email-add banner from Phase 8 to add one. Then verify it via OTP.

> If admin doesn't have an email, the smoke tests in the Setup Wizard can use any other test inbox you control.

## Deploy procedure (every release)

### Step 1 — Backup current state (~10 seconds)

1. Log in to XCloud as admin
2. Go to admin Users page
3. Click **💾 دانلود پشتیبان (.sql)** (added by Phase 12b)
4. Save the downloaded file somewhere safe

> If this is the first-ever deploy of v3.2 (Phase 12b doesn't exist in current production code), use phpMyAdmin's "Export" feature instead.

### Step 2 — Upload files via cPanel File Manager

1. Open cPanel → File Manager → navigate to your XCloud directory
2. Upload (or replace) these:
   - `index.php` (overwrite existing)
   - `lib/` directory:
     - `lib/.htaccess`
     - `lib/PHPMailer/PHPMailer.php`
     - `lib/PHPMailer/SMTP.php`
     - `lib/PHPMailer/Exception.php`
3. Verify file permissions: `index.php` should be `644`, `lib/` should be `755`

### Step 3 — Trigger migrations (~2 seconds)

Visit `https://xcloud.retrivo.ir/index.php` once in browser. Migrations run automatically on first load (idempotent).

### Step 4 — Run Setup Wizard (~3-5 minutes for first deploy, ~30 seconds for re-deploys)

1. Log in as admin
2. You'll see a blue banner on the Users page: **🛠️ Setup هنوز کامل نشده است**
3. Click **باز کردن Setup Wizard**
4. Follow the 6 sections in order:
   - **۱. پیش‌نیازها** — should be all green ✅. If any red, fix before continuing.
   - **۲. SMTP** — fill in or update credentials, save
   - **۳. تست SMTP** — enter your email, click "ارسال ایمیل تست", verify it arrives
   - **۴. Smoke tests** — for each of the 3 OTP types: send → check inbox → enter received code → verify ✓
   - **۵. ثبت‌نام عمومی** — click "فعال کردن" (or skip if you want to enable later)
   - **۶. قفل کردن Setup** — once all above ✓, click red **🔒 قفل کردن Setup**

### Step 5 — Smoke verify (~30 seconds)

From an incognito browser:
1. Visit homepage → click "ثبت‌نام در XCloud" link
2. Sign up with a real new email
3. Verify OTP arrives, complete registration
4. Auto-login to dashboard expected

If this works, deploy is **done**.

## Re-deploys (subsequent releases of v3.2.x)

After the first v3.2 deploy with the Setup Wizard, future deploys are even simpler:

1. **Backup** via admin Users page → 💾 button (5 seconds)
2. **Upload** changed files via cPanel (varies)
3. **Refresh** any open browser tab — that's it

The Setup Wizard is locked and stays locked unless you explicitly unlock by setting `setup_completed = '0'` in `settings` table via phpMyAdmin.

## Monitoring (first week after deploy)

### Daily checks

Run these queries in phpMyAdmin:

```sql
-- Failure rate last 24h (target: < 5%)
SELECT status, COUNT(*) AS count
FROM email_send_log
WHERE created_at > NOW() - INTERVAL 1 DAY
GROUP BY status;

-- Daily new registrations
SELECT DATE(created_at) AS day, COUNT(*) AS new_users
FROM users
WHERE email_verified = 1 AND created_at > NOW() - INTERVAL 7 DAY
GROUP BY DATE(created_at)
ORDER BY day DESC;

-- Suspicious IPs (>10 failed attempts/hour)
SELECT ip_address, COUNT(*) AS attempts
FROM activity_log
WHERE action_type IN ('login_failed', 'register_otp_requested')
  AND created_at > NOW() - INTERVAL 1 HOUR
GROUP BY ip_address
HAVING attempts > 10
ORDER BY attempts DESC;
```

### Disk usage

Monitor table sizes monthly:
```sql
SELECT table_name, ROUND(data_length/1024/1024, 2) AS mb
FROM information_schema.tables
WHERE table_schema = DATABASE()
  AND table_name IN ('email_otps', 'email_send_log', 'activity_log')
ORDER BY data_length DESC;
```

> The `maybeRunOtpCleanup()` from Phase 3 auto-deletes day-old OTPs and 30-day-old send logs. If tables grow unexpectedly, check that cleanup is running (see `otp_last_cleanup` setting).

## Rollback plan

If a critical issue appears post-deploy:

### Quick rollback (~30 seconds)
1. Go to admin Users page → click "غیرفعال کردن" on the registration toggle. New signups stopped.
2. Investigate the issue without rushing.

### Full rollback (~5 minutes)
1. cPanel File Manager → upload previous `index.php` (overwriting the new one)
2. Restore database from the `.sql` backup taken in Step 1:
   - phpMyAdmin → Import → upload the .sql file
3. The new tables (`email_otps`, `email_send_log`) and new columns can stay — they don't affect old code paths.

### Nuclear option — drop everything new
```sql
DROP TABLE IF EXISTS email_otps;
DROP TABLE IF EXISTS email_send_log;
ALTER TABLE users
  DROP COLUMN email,
  DROP COLUMN email_verified,
  DROP COLUMN email_verified_at;
DELETE FROM settings
WHERE setting_key LIKE 'smtp_%'
   OR setting_key LIKE 'otp_%'
   OR setting_key LIKE 'default_user_%'
   OR setting_key IN ('registration_enabled', 'setup_completed', 'otp_last_cleanup');
```
